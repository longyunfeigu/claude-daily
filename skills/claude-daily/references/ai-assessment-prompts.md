# AI 使用评估契约（ai-assessment mode）

供 `claude-daily ai-assessment` 流程使用。评分、缺陷根因追踪、reduce 全部按本文件执行。
评估的核心问题（PRD §2）：**是人领导 AI，还是 AI 领导人。** 不是看任务有没有完成，是看人在关键目标、边界、判断、验收、风险控制上是否保持主导。

## 通用铁律

贯穿 MAP 和 REDUCE 全程：

1. **证据优先**（PRD §6.1）：每个判断绑 transcript 证据；无证据写「证据不足」，不凭感觉打分。
2. **行为评估，非人身**（§6.2）：写「未发现验证命令」，不写「这个人不认真 / 能力差」。
3. **不唯结果**（§6.4）：复杂任务失败但设了边界 / 要求验证 / 及时止损 → AI 使用能力仍可能高；简单任务成功但全程无约束 / 无验证 / 无人工判断 → 不能给高分。
4. **不拿 AI 自述当事实**：AI 在 transcript 里说「测试通过了」不等于通过；以 `_assessment_context.json` 的 `commands[].outcome` 与 `claims[].prev_verify_outcome` 为准。这是这类评估的头号死法，务必守住。

## §A 输入契约

只读两份，**AI 回复正文（解释 / 代码）一律不读**（PRD §18.2）：

- `_assessment_context.json`（Plan 1 投影层产物），每 session：
  - `commands`：`[{cmd, kind(verify|explore|other), outcome(ok|fail|null)}]` —— 只有 `verify` 命令的 outcome 有意义；`explore`（grep/find/cd 等）非零退出是良性，outcome 为 null，**不算失败**。
  - `edit_events`：`[{file, n}]`，**不去重**——`n` 高 = 反复改同一文件（补丁套补丁的信号）。
  - `claims`：`[{claim, prev_verify, prev_verify_outcome}]` —— AI 完成声明 + 它之前最近一次验证命令及其成败。
  - `loops`：`{归一化命令: 连失次数}`，≥2 = 补丁套补丁。
  - `danger_ops`：`[{cmd}]` —— 真·高危操作（已排除 `/tmp` 使い捨て）。
  - `event_backbone`：`[{i, type, text}]`，全量事件骨架（`user`/`claim`/`verify`/`cmd`/`edit`/`tool_fail`），**不砍头尾**，用于定位锚点。
  - `user_prompts`：真人原话全文。
- `_context.json`：需要还原叙事时取 `compact_events`（可选）。

**证据类型 → 维度的绑定**（防止拿 A 类证据糊 B 维度）：

| 维度 | 主证据 |
|---|---|
| 目标与上下文 | `user_prompts`（首条 / 每轮开头）|
| 边界与约束 | `user_prompts` 的禁止项 + `danger_ops` / 越界编辑 |
| 任务拆解 | `user_prompts` 的阶段性 + 是否「先调研再做」|
| 主导权 | `event_backbone` 里 user 掰方向 / 拍板 + 中断 |
| 验证意识 | `commands`（verify 的 outcome）+ `claims[].prev_verify_outcome` |
| 纠错与止损 | `loops` + claim 失败后的走向 + user 喊停 |
| 风险意识 | `user_prompts` 对数据 / 安全 / 删除的要求 + `danger_ops` 有无确认 |
| 沉淀复用 | `edit_events` 是否写入规则 / 测试 / checklist 文件 |

## §B 单 session MAP 输出规范

每个 top-level session 产一份**结构化证据记录**。MAP 在 per-session subagent 里跑，只喂该 session 的投影记录。

### B.1 八维度评分（自包含评分块）

维度固定为 PRD §7 的 8 个，逐个按骨架写——**维度名 → 分数 → 紧跟它的直属证据链**：

```text
<维度名>：<0/3|1/3|2/3|3/3|N/A>
证据链（这条证明什么）：
- 材料：<user 原话 / commands 结果 / claims / edit_events / danger_ops，原话保留措辞，可截断不得改写>
- 支撑结论：<这些材料为什么撑这个分>
```

8 个维度：`目标与上下文`、`边界与约束`、`任务拆解`、`主导权`、`验证意识`、`纠错与止损`、`风险意识`、`沉淀复用`。

分档：`0/3` 明显缺失且已 / 很可能影响结果；`1/3` 有一点但不足以稳定降风险；`2/3` 基本到位有小缺口；`3/3` 强信号、行为稳定可复用；`N/A` 写清是「不适用」还是「证据不足」。

**禁止**退化成「分数堆顶部、材料堆底部」（读者看不出哪条证据撑哪个维度）。骨架的好 / 坏例子结构直接照搬 `references/report-prompts.md` 的「AI 使用评分」节。

关键维度（`主导权` / `验证意识` / `纠错与止损` / `风险意识`）必须按 §E 行为锚点判，不许往中间（2/3）挤。

### B.2 强制极值（治 central tendency）

每个维度的证据链必须挂**最强或最弱**那一条（标 `🔺最强` / `🔻最弱`），不要写「总体平衡」式总结——总结是区分力的杀手（PRD §18.8）。

### B.3 Layer 0.5 缺陷根因追踪

对每个**会上露头的缺陷**追一段。缺陷判据（任一触发）：

- `claims[].prev_verify_outcome == "fail"`（AI 声称完成，但其前最近验证是失败）
- `loops` 非空（同命令连失 ≥2）
- `danger_ops` 非空

每段按 PRD §18.5 六段写：

1. **缺陷**：什么坏了（产出质量 / 错误行为）。
2. **怎么暴露的**：`[@event_backbone 锚点]` 谁、怎么发现的。
3. **沟通根因**：哪句交互放它进来 + 缺了「目标 / 边界 / 验证 / 止损」中的哪一项。
4. **行为归因（非人身）**：归到「委派结构缺口」，不是「人菜」。
5. **能力信号**：缺口（引入缺陷那一下）+ 恢复方式（响应缺陷的动作）**都写**。
6. **下次护栏**：可照抄。

无缺陷写 `本 session 未发现会上露头的缺陷`（注意：仅指 session 内露头的 A 类缺陷；静默出厂的 B 类不在本评估范围）。

### B.4 置信度

本 session 整体置信度 `低 / 中 / 高` + 一句依据（如「`user_prompts` 仅 6 条，目标 / 上下文维度靠框架推断」）。

## §C 个人 REDUCE 规范

跨该人本批所有 session 聚合成人级画像。**只吃 MAP 产出的结构化记录，不重读 transcript。**

### C.1 不做算术平均（PRD §9.2 / §18.7）

- 关键维度（主导 / 验证 / 纠错 / 风险）取**下界 + 标波动**。例：纠错维度 A 为 `2`、B 为 `1` → 写 `1-2 不稳定`，**不取均值 1.5**（均值会把「连失 5 次的循环」洗白成「基本到位」）。
- 其余维度可给区间（如 `2-3，视任务而定`）。

### C.2 红线压顶（PRD §9.3）

任一 session 命中以下红线 → **总体等级最高 L1**，除非后续 session 有明确纠正：

- 高风险修改无验证、无确认。
- 明显把架构 / 安全 / 产品关键取舍交给 AI 决定。
- AI 多次跑偏，用户没发现或没止损。
- 生成内容进入项目但没有读 diff、测试或 review 证据。

可上调信号：用户主动质疑 AI 并要证据 / 发现风险并暂停危险操作 / 把失败转成规则·测试·checklist / 帮团队沉淀可复用工作流。

### C.3 置信度传播

样本少 / `user_prompts` 多为空 / 只有闲聊无工具证据 → 必须标 `低置信，仅供人工复核`，不给干净等级。

### C.4 等级

给 L0-L4（PRD §8）+ 一句话定性，**不替代维度证据**。等级靠关键维度（§C.1）+ 红线（§C.2）得出，不靠平均。

- <具体到命令 / 动作>

## 可复用 checklist（存起来）

## §E 行为锚点标尺

治「谁都打 2/3」。4 个关键维度各给 `3/3` vs `1/3` 的**具体行为锚点**（"长这样才算 3，长那样只能 1"）；2/3 是两者之间有明显小缺口。

**主导权**
- `3/3`：关键处掰方向且有据——质疑 AI 方案 / 要求回到代码事实核验 / 在 AI 动手前喊停对齐（`event_backbone` 有 user 掰方向 + 中断）。
- `1/3`：基本照单全收 AI 方案，无追问 / 无拍板；架构 / 依赖 / 范围等关键取舍由 AI 默认决定。

**验证意识**
- `3/3`：完成 claim 前紧挨真实 test/build/lint 且 outcome 可查；失败如实报（`claims` 的 `prev_verify_outcome` 多为 `ok` 且有真命令）。
- `1/3`：AI 说「完成 / 通过」但 `commands` 无对应 verify，或 claim 的 `prev_verify_outcome` 为 `null`；新依赖 / 高危改动无核查。

**纠错与止损**
- `3/3`：AI 跑偏 / 失败时早抓快止——`loops` 为空，或失败后有定向排查（grep 查根因）再改；user 喊停 / 回滚。
- `1/3`：`loops` 非空（同命令连失 ≥2 补丁套补丁），或同区域反复 edit（`edit_events` 的 `n` 高）无收敛；user 未发现明显越界 / 幻觉。

**风险意识**
- `3/3`：对数据 / 安全 / 删除 / 迁移设确认点；高危操作前要求核查（`danger_ops` 有确认，或先在 throwaway 上试）。
- `1/3`：高危任务无 review / 测试 / 备份；`danger_ops` 直接执行无确认；只关注「能不能做」不关注「会不会埋坑」。
