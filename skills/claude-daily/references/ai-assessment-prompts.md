# AI 使用评估契约（ai-assessment mode）

供 `claude-daily ai-assessment` 流程使用。
评估的核心问题（PRD §2）：**是人领导 AI，还是 AI 领导人。**

## 输出哲学（最重要的一条）

**给证据和叙事，不给数字分，不给能力等级。**

理由：per-dimension 的 `0-3` 分与 `L0-L4` 等级都是 AI 摇摆的序数判断——假精度、跑两遍会变、招来「平均 / 排名 / 当人的能力结论」这类 PRD §4 非目标与 §9.2「不要平均」明令禁止的滥用；而我们在真实数据上验证过：**真正的区分力来自行为证据本身（loops、claim↔验证、截图反查、安全取舍裸奔），不来自数字**。所以评估产出 =
逐回合执行叙事 + 缺陷根因 + 绑证据的定性判断（强 / 到位 / 有缺口 / 证据不足）+ 触发旗标 + 风险/培训主题。

> 本节取代 PRD §8（L0-L4 等级）与 §9.1（0-3 维度分）在评估**输出**上的用法；§7 的 8 个维度名仍用，§9.3 的红线规则改为驱动「旗标」。

## 通用铁律

1. **证据优先**（PRD §6.1）：每个判断绑 transcript 证据；无证据写「证据不足」。
2. **行为评估，非人身**（§6.2）：写「未发现验证命令」，不写「这个人不认真」。
3. **不唯结果**（§6.4）：复杂任务失败但设了边界 / 验证 / 止损 → 仍可能是强协作；简单任务成功但全程裸奔 → 不算强。
4. **不拿 AI 自述当事实**：以 `commands[].outcome`、`claims[].prev_verify_outcome` 为准，不信 AI 嘴上「完成了」。
5. **不打数字分、不给 L 等级**：判断一律用「强 / 到位 / 有缺口 / 证据不足」并紧绑证据；管理用途靠**旗标 + 风险主题** triage，不给能力排名。

## §A 输入契约

只读两份，**AI 回复正文一律不读**（PRD §18.2）：

- `_assessment_context.json`（投影层产物），每 session：`commands[{cmd,kind,outcome}]`、`edit_events[{file,n}]`（不去重）、`claims[{claim,prev_verify,prev_verify_outcome}]`、`loops{cmd:连失次数}`、`danger_ops[{cmd}]`、`event_backbone[{i,type,text}]`（全量，定位锚点用）、`user_prompts`。
- `_context.json`：取 `compact_events` 还原叙事（可选）。

证据类型 → 维度绑定（防止拿 A 类证据糊 B 维度）：

| 维度 | 主证据 |
|---|---|
| 目标与上下文 | `user_prompts`（首条/每轮开头）|
| 边界与约束 | `user_prompts` 禁止项 + `danger_ops`/越界编辑 |
| 任务拆解 | `user_prompts` 阶段性 + 是否先调研再做 |
| 主导权 | `event_backbone` 里 user 掰方向/拍板 + 中断 |
| 验证意识 | `commands`(verify 的 outcome) + `claims[].prev_verify_outcome` |
| 纠错与止损 | `loops` + claim 失败后走向 + user 喊停 |
| 风险意识 | `user_prompts` 对数据/安全/删除的要求 + `danger_ops` 有无确认 |
| 沉淀复用 | `edit_events` 是否写入规则/测试/checklist 文件 |

## §B 单 session MAP（执行叙事优先）

每个 top-level session 产一份记录，三段 + 置信度。

### B.1 执行叙事（逐回合，这是 body，不是总结）

用 `event_backbone` + `user_prompts` 还原**任务怎么被一步步驱动的**。每个实质回合一条：

```
[@锚点] 用户这轮要什么（引原话）→ AI 怎么回 / 做了什么 → 这轮落地了什么（命令成败 / 编辑 / 验证）→ 命名的 pushback（如有）
```

- 覆盖主要回合，**不要压成三五条干 bullet**；标出弧线里的关键转折（掰方向 / 中断 / 撞坑 / 止损 / 截图反查）。
- 一个有 10+ 回合的 session 不应只剩 4 条。这一段展示的是过程轨迹，不是结论。
- 薄 session（prompts 少、无工具证据）：叙事一两句，写明「只读问答 / 闲聊，无可评估的协作行为」。

### B.2 缺陷根因追踪（内嵌叙事，触发即追）

触发（任一）：`claims[].prev_verify_outcome=="fail"` / `loops` 非空 / `danger_ops` 非空。
六段（PRD §18.5）：缺陷 → 怎么暴露 `[@锚点]` → 沟通根因（缺了目标/边界/验证/止损中哪项）→ 行为归因（非人身，归到委派结构缺口）→ 能力信号（缺口 + 恢复都写）→ 下次护栏（可照抄）。
无则一句 `本 session 未发现会上露头的缺陷`。

### B.3 维度判断（8 维，定性 + 证据，无数字）

8 维（PRD §7）逐个给，**不打分**：

```
<维度>：<强 | 到位 | 有缺口 | 证据不足>
- 直属证据：<引 user 原话 / 命令成败 / claims / edit_events，标 🔺最强 或 🔻最弱>
```

8 维：`目标与上下文`、`边界与约束`、`任务拆解`、`主导权`、`验证意识`、`纠错与止损`、`风险意识`、`沉淀复用`。
按 §E 判据。关键维度（主导/验证/纠错/风险）必须有**行为证据**，不凭 prompt；无证据写「证据不足」，不硬判。

### B.4 置信度

本 session 整体 `低 / 中 / 高` + 一句依据。

## §C 个人 REDUCE（旗标 + 主题，不评级不平均）

跨该人本批所有 session 聚合。**不给等级、不平均、不排名。** 产出：

- **一句话判断**（绑证据）：在可评估的 session 里，这个人是领导 AI 还是被带走；最强项 + 最一致的短板。
- **触发旗标**（给管理 triage，可多选）：`红线命中` / `证据不足-需人工复核` / `本期无明显风险`。
- **强项主题**：跨 session 反复出现的好动作（绑证据）。
- **风险主题**：跨 session 反复出现的坏味道/缺口（绑证据 + 为什么伤结果）。
- **培训主题**：从风险主题推出的可练动作。
- **置信度 + 不应过度解读**：样本薄 / `user_prompts` 多空 → 标低置信、说清哪些结论不可外推。

**红线**（命中即打 `红线命中` 旗标，PRD §9.3）：高危改动无验证/确认；架构·安全·产品关键取舍交给 AI；AI 多次跑偏无止损；生成内容进项目无 diff/test/review。

**看模式不看单次**（取代旧的「取下界」——没有数字可取）：一个风险只在 1 个 session 出现 → 标「单次，可能任务差异」；多 session 复现 → 「稳定模式」。

## §D 三输出文件契约

### D.1 `_ai_assessment.personal.md`（形态 B，本人看，叙事优先）

```markdown
# 你的 AI 使用复盘 · <样本=N session>

## 一、各 session 执行叙事
<每个 session：B.1 执行叙事 + B.2 内嵌缺陷根因 + B.3 维度判断（定性）>

## 二、做得好的（保持）
- <绑具体动作 + 效果，引 user 原话>

## 三、容易埋坑的（改这里）
- <绑证据 + 为什么伤结果>

## 四、下次可以这样问（照抄，基于本人原话改写）

## 五、下次必须验证什么

## 六、可复用 checklist
```

深引、多引原话。**全文无数字分、无等级。**

### D.2 `_ai_assessment.manager.md`（形态 A，负责人看，全透明、可点击复核）

```markdown
# AI 使用评估 · <成员> · <时间范围 / 样本数>

旗标：<红线命中 / 证据不足-需人工复核 / 本期无明显风险>   置信度：<...>

## 一句话判断

## 能力维度
| 维度 | 判断 | 证据摘要 | 风险/改进 |
|---|---|---|---|
（判断列只填 强/到位/有缺口/证据不足，无数字）

## 项目风险

## 培训建议

## 不应过度解读的部分
```

按 PRD §11 全透明审计：可含原始证据，Layer 0.5 可摘一句 + 指针、也可深引；锚点用纯 `@<整数>` 格式、每段声明所属 session（网页按 `@(\d+)` 解析并定位到该 session 的 `event_backbone` 渲染可点击弹窗）；**无总体等级、无数字**。正文仍以判断 + 摘要为主、原始细节走锚点下钻（可读性，非披露限制）。

## §E 定性判据

取代行为锚点打分。4 个关键维度各给「什么样算 强 / 到位 / 有缺口 / 证据不足」：

**主导权**
- 强：关键处掰方向有据（质疑方案 / 要求回到代码事实 / 动手前喊停对齐），有中断或据实拍板。
- 到位：基本保持主导，偶有照单全收。
- 有缺口：架构 / 依赖 / 范围等关键取舍多由 AI 默认决定，少追问。
- 证据不足：无足够交互可判（如只读问答）。

**验证意识**
- 强：完成主张前紧挨真实 test/build/lint 且 outcome 可查，失败如实报。
- 到位：有验证但不稳定 / 有遗漏。
- 有缺口：AI 说完成但无对应 verify，或新依赖 / 高危改动无核查。
- 证据不足：本 session 无 claim 也无可评 verify 行为。

**纠错与止损**
- 强：AI 跑偏 / 失败时早抓快止，`loops` 为空或失败后定向排查再改。
- 到位：能纠错，但出现过一次收敛慢。
- 有缺口：`loops` 非空（连失 ≥2 补丁套环）或同区域反复 edit 无收敛。
- 证据不足：本 session 无失败 / 越界事件可评。

**风险意识**
- 强：对数据 / 安全 / 删除 / 迁移设确认点，高危前要求核查。
- 到位：操作层风险可控，但某类风险（如安全含义）未显式处理。
- 有缺口：高危任务无 review / 测试 / 备份，或危险操作直接执行无确认。
- 证据不足：本 session 无涉及风险的动作。
