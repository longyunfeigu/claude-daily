# AI 使用能力评估 PRD

## 1. 背景

`claude-daily` 当前定位是把本地 Claude Code 会话记录整理成每日工作报告，并在 personal Session Card 中附带 `人机协作` 与 `AI 使用评分`。这已经能看见一部分 AI 使用行为，但它仍然是日报工具，不是面向团队负责人的 AI 使用能力评估工具。

团队负责人需要了解团队成员使用 AI 的意识和能力：谁能主导 AI、谁容易被 AI 带着走、哪些行为会给项目埋坑、团队应该补什么训练。这个需求不能只靠“任务有没有完成”判断，因为复杂任务失败不等于 AI 用得差，简单任务成功也不等于 AI 用得好。

本 PRD 定义一个新的评估能力：基于 Claude Code transcript 中可观察到的证据，评估成员是否表现出主导 AI、约束 AI、验证 AI、纠正 AI、沉淀 AI 使用经验的行为。

## 2. 核心判断

评估的核心问题是：

> 是人领导 AI，还是 AI 领导人。

“人领导 AI”不是指人手写所有细节，而是指人在关键目标、边界、判断、验收、风险控制上保持主导。

“AI 领导人”的典型风险是：用户把目标、方案、架构、取舍、验收都交给 AI 猜，AI 生成的内容未经验证进入项目，短期看似提速，长期在代码质量、架构边界、安全、维护成本上埋坑。

## 3. 目标

### 3.1 产品目标

- 帮助团队负责人了解成员 AI 使用意识与能力。
- 帮助个人知道自己下一步该如何改进 prompt、委派方式、验证习惯和风险控制方式。
- 帮助团队发现共性培训需求。
- 帮助识别项目风险：哪些工作流中 AI 输出可能未经足够约束或验证。
- 支持谨慎的能力考核，但不把 transcript 评分当作人的完整能力结论。

### 3.2 用户目标

团队负责人希望看到：

- 每个人当前 AI 使用能力大致处于什么水平。
- 这个判断来自哪些可复核证据。
- 主要风险在哪里。
- 下一步应该训练什么。
- 哪些结论证据不足，不能过度解读。

个人成员希望看到：

- 哪些 prompt 行为有效。
- 哪些委派方式容易让 AI 跑偏。
- 哪些任务缺少验证或人工判断。
- 下次可以照抄的改进模板。

## 4. 非目标

- 不评估成员真实思考能力的全貌。transcript 只能看到被记录下来的行为。
- 不把一次 session 的结果直接等同于个人能力。
- 不用 token 数、会话数量、工具调用次数直接代表能力。
- 不做没有证据支撑的人格化判断，如“这个人不认真”“这个人能力差”。
- 不鼓励为了评分而写表演式 prompt。
- 不替代 code review、架构评审、安全审查、绩效面谈。

## 5. 适用范围

### 5.1 输入

主要输入来自 Claude Code 本地记录：

- 用户 prompt。
- AI 回复摘要或片段。
- 工具调用。
- 命令执行记录。
- 文件读写记录。
- Web 查询记录。
- session 时间、项目、持续时长。
- `ccusage` token 统计，仅作为使用量背景，不作为能力评分核心依据。

### 5.2 输出

新增三类评估输出：

- 个人 AI 使用能力画像。
- 团队负责人评估报告。
- 团队培训与项目风险摘要。

## 6. 评估原则

### 6.1 证据优先

所有判断必须绑定 transcript 证据。没有证据时写“证据不足”，不能凭感觉打分。

### 6.2 行为评估，不做人身判断

评估对象是可观察行为，例如“没有发现验证命令”“关键架构取舍由 AI 直接给出且未见用户确认”，而不是“这个人不会思考”。

### 6.3 置信度必须显式给出

每个成员和每个维度都要给置信度：低 / 中 / 高。

低置信度常见原因：

- session 太少。
- user prompts 为空或过短。
- 只有聊天，没有工具证据。
- 工作成果发生在线下或其他系统。
- transcript 缺失关键上下文。

### 6.4 结果导向但不唯结果

评估 AI 使用方式，不直接按任务成败评分。

例如：

- 复杂任务失败，但用户设了边界、要求验证、及时止损，AI 使用能力可能仍然高。
- 简单任务成功，但全程无约束、无验证、无人工判断，AI 使用能力不能给高分。

### 6.5 管理用途要更保守

给团队负责人看的报告应突出风险、培训需求和证据，不输出羞辱性、标签化、不可申诉的结论。

## 7. 评估维度

### 7.1 目标与上下文

看用户是否说明要解决什么问题、背景是什么、成功标准是什么。

强信号：

- 明确说明业务目标或技术目标。
- 给出相关文件、模块、现状、上下文。
- 说明为什么要做，而不只是让 AI 开始做。

风险信号：

- 只说“帮我做一下”“修一下”“优化一下”。
- AI 需要大量猜测上下文。
- session 中多次返工来自目标不清。

### 7.2 边界与约束

看用户是否说明范围、禁止项、兼容性、风格、不能改的地方。

强信号：

- 明确“不要改 X”“只处理 Y”“先不要实现，只分析”。
- 给出输出格式、文件范围、依赖限制。
- 高风险操作前要求确认。

风险信号：

- AI 自行扩大范围。
- 未经确认引入依赖、重构架构、改动无关文件。
- 用户事后才发现 AI 越界。

### 7.3 任务拆解

看用户是否把大任务拆成探索、方案、决策、实现、验证。

强信号：

- 先要求 AI 调研现状，再出方案。
- 实现前要求列风险、权衡或验证路径。
- 大任务中设置 checkpoint。

风险信号：

- 一上来整包委派“你直接做完”。
- AI 边猜边改，没有阶段性确认。
- 多轮补丁堆叠但没有重新校准目标。

### 7.4 主导权

看关键判断是否由人保持主导。

强信号：

- 用户要求 AI 给选项和依据，然后自己拍板。
- 用户挑战 AI 结论，要求回到代码、文档或事实核验。
- 产品、架构、安全、上线风险等关键取舍有人工确认。

风险信号：

- AI 直接决定架构、依赖、范围、产品取舍。
- 用户只接受 AI 方案，没有追问或确认。
- 高影响决策没有人工 checkpoint。

### 7.5 验证意识

看用户是否要求或接受验证，以及 AI 是否实际执行验证。

强信号：

- 要求跑测试、lint、build、最小复现、文档核查。
- 要求读 diff 或总结改动风险。
- 对无法验证的部分明确标注“未验证”。

风险信号：

- AI 说“完成了”后没有命令或证据。
- 新依赖、新 API、新安全相关改动没有核查。
- 报告声称通过验证，但 transcript 无证据。

### 7.6 纠错与止损

看 AI 跑偏、出错、越界时，用户是否及时发现并收敛。

强信号：

- 用户及时指出“不对”“不要这个”“回滚”“先停”。
- AI 或用户执行回滚、缩小范围、重新确认。
- 同类错误没有无限循环。

风险信号：

- 错误发生后继续补丁套补丁。
- 同一区域反复修改但无明确收敛。
- 用户没有发现明显越界或幻觉。

### 7.7 风险意识

看用户是否识别项目风险，而不是只追求生成速度。

强信号：

- 对数据、安全、权限、兼容性、性能、维护成本提出要求。
- 对删除、迁移、批量修改、发布等高风险动作设置确认点。
- 要求 AI 暴露不确定性和剩余风险。

风险信号：

- 高风险任务没有 review、测试或备份。
- AI 建议危险操作，用户未要求核查。
- 只关注“能不能做”，不关注“会不会埋坑”。

### 7.8 沉淀复用

看用户是否把经验转化成可复用资产。

强信号：

- 把有效 prompt、规则、测试、文档、checklist 沉淀下来。
- 发现问题后更新团队规范。
- 把一次纠错转化为下次的防线。

风险信号：

- 同类问题多次重复出现。
- 没有记录有效用法或失败教训。
- 每次都从零开始依赖 AI 猜。

## 8. 能力等级

能力等级用于概括，不替代维度证据。

### L0：被 AI 牵着走

特征：

- 需求模糊。
- 缺少边界。
- 关键判断外包给 AI。
- 缺少验证。
- AI 跑偏后没有及时止损。

典型风险：

- 代码或方案看似完成，实际隐藏质量、架构、安全或维护风险。

### L1：会操作 AI

特征：

- 能让 AI 完成一些任务。
- 偶尔能给上下文或约束。
- 验证和纠错不稳定。
- 对 AI 输出的采纳偏粗粒度。

典型风险：

- 简单任务效率提升明显，但复杂任务容易被 AI 带偏。

### L2：合格协作者

特征：

- 能讲清目标和范围。
- 知道让 AI 先调研或给方案。
- 基本会要求验证。
- 关键位置有人工判断。

典型风险：

- 还需要提升系统性拆解、风险识别和沉淀能力。

### L3：强主导者

特征：

- 能拆任务、设 checkpoint、挑战 AI。
- 能控制范围和风险。
- 能发现 AI 幻觉、越界和过度设计。
- 验证习惯稳定。

典型价值：

- 能把 AI 作为加速器，而不是让 AI 接管判断。

### L4：团队放大器

特征：

- 不仅自己会用，还能沉淀流程、prompt、模板、规范。
- 能识别团队共性问题并改进工作流。
- 能带动他人提升 AI 使用质量。

典型价值：

- 提升团队整体 AI 协作水平，降低项目级 AI 风险。

## 9. 评分规则

### 9.1 维度评分

每个维度使用：

- `0/3`：明显缺失，且已经或很可能影响结果。
- `1/3`：有一点体现，但不足以稳定降低风险。
- `2/3`：基本到位，少量缺口。
- `3/3`：强信号，行为稳定且可复用。
- `N/A`：不适用或证据不足。

### 9.2 总体等级

总体等级不做简单平均。优先看关键维度：

- 主导权。
- 验证意识。
- 纠错与止损。
- 风险意识。

如果这些关键维度低，即使目标表达很好，也不能给高等级。

### 9.3 红线规则

出现以下证据时，总体等级最高不超过 L1，除非后续 session 有明确纠正：

- 高风险修改无验证、无确认。
- 明显把架构、安全、产品关键取舍交给 AI 决定。
- AI 多次跑偏，用户没有发现或没有止损。
- 生成内容进入项目但没有读 diff、测试或 review 证据。

出现以下证据时，可以上调：

- 用户主动质疑 AI 并要求证据。
- 用户发现风险并暂停危险操作。
- 用户把失败转化为规则、测试或 checklist。
- 用户帮助团队沉淀可复用工作流。

## 10. 报告形态

### 10.1 团队负责人报告

报告应以行动为中心，不展示过多原始聊天内容。

推荐结构：

```markdown
# AI 使用能力评估

时间范围：
成员：
样本数量：
总体等级：
置信度：

## 一句话判断

## 主要证据

## 能力维度

| 维度 | 评分 | 证据摘要 | 风险/改进 |
|---|---:|---|---|

## 项目风险

## 培训建议

## 不应过度解读的部分
```

### 10.2 个人改进报告

个人报告应更具体，给可执行改进建议。

推荐结构：

```markdown
# AI 使用改进建议

## 做得好的地方

## 容易埋坑的地方

## 下次可以这样问

## 下次必须验证什么

## 可复用 prompt / checklist
```

### 10.3 团队汇总报告

团队报告用于发现整体培训需求。

推荐结构：

```markdown
# 团队 AI 使用能力概览

时间范围：
成员数：
session 数：

## 等级分布

## 共性强项

## 共性风险

## 培训优先级

## 需要人工复核的成员或项目
```

## 11. 隐私与管理边界

### 11.1 默认最小披露

给团队负责人看的报告默认不展示完整 prompt、完整代码、敏感文件路径。只展示必要证据摘要。

### 11.2 原文引用限制

只有在支撑关键判断时才引用用户原话。引用应短、必要、可复核。

### 11.3 申诉与复核

当结果用于考核时，必须允许成员查看证据来源并补充上下文。

### 11.4 不确定性表达

证据不足时必须写：

- `证据不足，不能判断`。
- `低置信度，仅建议人工复核`。
- `该结论只反映 Claude Code transcript 中可见行为`。

## 12. 产品方案

### 12.1 新增模式

在现有 `claude-daily` 基础上新增 AI 使用评估模式：

- `跑 claude-daily ai-assessment`
- `生成 AI 使用能力评估`
- `生成团队负责人版 AI 使用评估`

### 12.2 输出文件

建议新增：

- `_ai_assessment.personal.md`
- `_ai_assessment.manager.md`
- `_ai_assessment.team.md`
- `_ai_assessment.meta.json`

### 12.3 与现有日报关系

AI 使用评估不应塞进普通日报正文里。普通日报回答“今天做了什么”，AI 使用评估回答“这个人如何使用 AI，以及有什么风险和训练需求”。

现有 personal Session Card 中的 `人机协作` 可以保留，但应作为评估模式的数据基础或轻量版本。

## 13. MVP 范围

第一版只做单人成员、单日或多日评估。

MVP 必须包含：

- 8 个维度评分。
- L0-L4 总体等级。
- 证据链。
- 置信度。
- 团队负责人报告。
- 个人改进建议。
- 项目风险。
- 培训建议。
- 证据不足保护规则。

MVP 不做：

- 排行榜。
- 自动绩效结论。
- 跨团队对标。
- 复杂可视化看板。
- 自动读取非 Claude Code 数据源。

## 14. 验收标准

### 14.1 准确性

抽样 10 个真实 session 人工复核：

- 关键判断都能在 transcript 中找到证据。
- 没有把 AI 自述当成事实。
- 没有把任务成败直接等同于 AI 使用能力。
- 证据不足时没有硬评。

### 14.2 可用性

团队负责人读完报告后，能回答：

- 这个人主要风险是什么。
- 下一步该训练什么。
- 哪些结论可信，哪些要人工复核。

个人读完报告后，能回答：

- 下次 prompt 应该补什么。
- 哪些任务必须设 checkpoint。
- 哪些输出必须验证。

### 14.3 安全性

- manager 报告不泄露不必要的敏感原文。
- 低置信度结论被明确标注。
- 用于考核时保留证据复核路径。

## 15. 实施计划

### 阶段一：人工样本验证

目标：验证评估框架是否能稳定区分“人领导 AI”和“被 AI 领导”。

动作：

- 选 3-5 个真实 session。
- 人工按 8 个维度评分。
- 标记争议点和证据不足点。
- 调整维度定义和红线规则。

### 阶段二：Prompt 化

目标：把评估规范写进 skill references。

动作：

- 新增 `references/ai-assessment-prompts.md`。
- 新增 `references/ai-assessment-review-checklist.md`。
- 明确 manager / personal 两种输出契约。

### 阶段三：脚本与产物接入

目标：在现有 context 生成基础上新增评估产物。

动作：

- 复用 `prepare.py` 和 `_context.json`。
- 新增评估步骤。
- 新增评估输出文件。
- 新增 review gate。

### 阶段四：团队汇总

目标：从多人、多日评估中生成团队负责人视图。

动作：

- 聚合成员等级、共性风险、培训需求。
- 控制隐私披露。
- 标记低置信度成员。

## 16. 开放问题

- 团队负责人报告是否允许引用用户原话，还是只给脱敏摘要？
- 评估时间窗口默认按日、周，还是按最近 N 个 session？
- 考核场景下，成员是否必须先看到个人报告并有机会补充上下文？
- 是否需要项目风险按 repo / project 聚合？
- 是否需要把 code review 结果、PR 结果、线上事故等外部数据接入评估？

## 17. 推荐下一步

先不要直接实现完整功能。下一步应做一个低成本验证：

1. 选 3-5 个真实 Claude Code session。
2. 按本 PRD 的 8 个维度人工评估。
3. 判断报告是否能帮团队负责人看出风险和培训需求。
4. 判断个人是否能从建议中改进下一次 prompt。
5. 再决定 prompt、脚本和输出结构怎么改。

## 18. 设计：缺陷根因追踪与分层解读

> 本章是 §15 阶段一「人工样本验证」的产出。已在本机真实 session 上验证：8 维度信号可从 transcript 取到，新方法能拉开旧方法（per-session 绝对打分）糊掉的差异。本章把验证过的方法固化为设计，供阶段二 prompt 化与阶段三脚本接入使用。

### 18.1 验证结论：为什么旧方法「区分不开人」

旧的 per-session 绝对打分区分不开人，根因不是数据源，是方法。四个具体原因：

- **LLM 评分往中间挤（central tendency）**：不给行为锚点，让它打 0-3，谁都是 2/3。「基本到位 / 有一点」这种抽象档位，AI 自己都分不清边界。
- **单 session 样本太小**：日常 session 又短又像，多个维度根本没东西可打。区分力来自跨 session 的模式，不是单次。
- **评分混进日报被稀释**：日报主语是「今天做了什么」，评分是附属字段，注意力和篇幅都给了前者。
- **缺相对参照/校准**：绝对打分对 LLM 难，相对判断（A 和 B 谁更主导）对 LLM 易，但现在没有任何 calibration。

修复这四点的是：**行为锚点标尺 + 人级聚合 + 强制找极值 + 事件流输入**（见 18.3–18.8）。与「是不是做成新 skill」无关——换打包不解决区分力。

### 18.2 输入边界：事件流，不是全文 transcript

判断只需要两类东西，AI 回复的散文（解释、代码正文）一律丢弃：

- **AI 做了什么**：tool calls（调了什么、动了哪些文件、跑没跑验证）——结构化事件，一行一个。
- **AI 声称了什么**：「完成了 / 测试通过了」——要被 transcript 反驳的主张。

正确输入 = `user prompts（意图/边界全文）+ tool 调用序列 + 命令成败 + AI 完成声明`。

控上下文的机制是 **per-session subagent fan-out**：每个 session 在自己的上下文窗口里被读，只产出 ~300-500 token 的结构化证据记录；汇总时评判只读 N 份证据记录（N×400 token），不读 N 份 transcript。上下文从几十万 token 降到几千。这正是现有 SKILL.md step 4 review gate 已在用的模式。

### 18.3 数据投影层（assessment projection）

**不重写 `prepare.py`**（会丢掉 token 去重、`ccusage` 对账、跨天切分、member_id 冲突处理等已测能力）。加一个评估专用投影层，消费同一份原始解析，只补 4 个信号——它们恰好卡住最能区分人的维度：

| 信号 | 现状缺口 | 为什么对评估致命 |
|---|---|---|
| 命令成败 / 退出码 | `commands` 只存命令字符串，`toolUseResult` 的 stdout / is_error 丢了 | 验证维度命门：只能看到「跑了 pytest」，看不到「过没过」，无法区分真验证和 AI 嘴上通过 |
| 编辑频次（不去重） | `files_modified` 是 set | 同文件改 9 次（补丁套补丁）和改 1 次长得一样，「纠错与止损」维度直接瞎 |
| claim↔验证关联 | 靠 LLM 从时间线现拼 | 现拼就拼错 → 退化成「拿 AI 自述当事实」这个头号 bug |
| 中段覆盖 | 只留头尾（assistant 最后 8 条 / compact_events 前 400 条） | 止损 / 跑偏发生在中段，长 session 中段没了，评估正好丢在关键处 |

两个**必须内置的去误判规则**（验证中实际踩到）：

- **区分「验证命令真失败」和「探索命令良性非零」**：`grep` no-match 返回 1、`cd && echo && find` 复合命令的非零退出，都不是验证失败。`exit≠0 ≠ 失败`。
- **danger 判定看路径上下文**：`rm -rf` 在 `/tmp` 使い捨て = 无害；且正则别匹配文件路径里的关键词（`alembic/versions` 曾被误标）。

投影层只在 `ai-assessment` mode 跑，日报路径完全不动。

### 18.4 三层解读结构

| 层 | 内容 | 给谁 | 隐私 |
|---|---|---|---|
| **Layer 0** 执行叙事 | 单 session「任务怎么被一步步驱动」+ 真证据（原话 + @锚点） | 证据底料 | 含原话，仅本人 |
| **Layer 0.5** 缺陷根因追踪 | 从一个缺陷倒推：哪里错 → 哪句交互放它进来 → 缺了什么 → 护栏 | 个人深引 / 管理摘要 | 个人深引；管理只摘一句 |
| **Layer 1** 人级解读 | A 管理向 / B 个人向判断，**引用 Layer 0/0.5 的片段当证据** | 管理者 / 本人 | A 守 §11 最小披露 |

铁律：上层判断不许凭空下结论，必须引用下层的具体片段。Layer 0/0.5 能从 transcript 半自动生成。

### 18.5 Layer 0.5 缺陷根因追踪（本章核心）

PRD 核心问题「人领导 AI vs AI 领导人」在**缺陷发生的瞬间最暴露**——一个 bug 出厂，就是「AI 把人带偏、人没接住」的最硬证据。Layer 0.5 从一个缺陷倒推它是在哪次人机交互、因为什么沟通方式溜进来的。

固定结构：

1. **缺陷**：什么坏了（产出质量 / 错误行为）。
2. **缺陷怎么暴露的**：`[@锚点]` 谁、怎么发现的（看实物 / 报错 / 验证失败）。
3. **沟通根因**：哪句交互放它进来 + 缺了什么（目标 / 边界 / 验证 / 止损 中的哪一项）。
4. **行为归因（非人身）**：归到「委派结构缺口」，不是「人菜」（守 §6.2）。
5. **这暴露的 AI 使用能力**：缺口（引入缺陷那一下）+ 恢复方式（响应缺陷的动作）都要看。
6. **下次的护栏**：可照抄。

**缺陷分两类，决定可行性与边界：**

- **A 类·会上露头**：session 里测试红 / 报错 / AI 说修好但验证失败 / 用户当场说不对。transcript 有痕迹，**MVP 只做这类**。
- **B 类·静默出厂**：当时没发现，事后 commit 回滚 / PR review / CI / 线上事故才暴露。transcript 零痕迹，须接外部数据 → 见 §4 非目标第 6 条、§16 开放问题。**MVP 不做，列为后续扩展。**

**已验证的真实例子（A 类）**：`vj-work` 交付的登录页功能全过、UI 很丑。沟通根因不是「人不懂前端」，而是**委派的 done-criteria 对定性质量是盲的**——验证全是二值行为判据（账号数、状态码、起得来、有截图），没有任何一条会因为「页面空 / 丑」变红，AI 于是精确优化了写进判据的东西、牺牲了没写进去的「设计质量」。这是结构性缺口，换谁跑同一 skill 都会掉进去。

### 18.6 两端对照：镜头靠什么区分人

**区分人的不是「出没出 bug」，是出 bug 之后的响应。** 同一缺陷（AI 交付丑 UI，设计意图没进 done 判据），两端：

| 镜头读的信号 | 强主导路径（真实 session） | 被 AI 带走路径（反演·非真人） |
|---|---|---|
| ① 有没有看真实产出 | 启动服务**亲眼看** → 才发现丑 | 接受 AI「测试全过、done」**没看实物** |
| ② 缺陷暴露后第一反应 | 「**根本原因是什么**，不要打补丁」 | 「行吧，那你**再弄漂亮点**」或直接收下 |
| ③ 修复落点 | 改 **skill 本体**（加 UI QA policy / 正向存在性判据 / 独立 review） | 改**这一个页面**，或重 prompt 让 AI 再 roll 一次 |
| ④ 结果 | 缺陷类**被永久消灭**（沉淀） | 同类缺陷**下次必复发**，或**直接出厂** |
| 命中维度 | 验证✅ 纠错✅ 主导✅ 沉淀✅ | 验证❌ 纠错❌ 主导❌ → §9.3 红线：生成内容进项目无独立 review |
| 等级判读 | L3→L4 | L1 / L0 |

镜头读的四个信号**全部是 transcript 可观测的**：启动命令（看没看实物）、用户反应文本（逼不逼根因）、编辑落点（改 skill 还是单文件）、有没有独立 review。这把 §6.4「不唯结果」落到了实处——读的是**对缺陷的响应**，不是缺陷本身。

### 18.7 reduce 规则：人级 / 团队级聚合

**Map（单 session）只产结构化证据，不产散文。** 每 session 输出：8 维度 0-3 / N/A + 每维度证据链 + 置信度 + 命中红线。

**Reduce 两条铁律：**

- **不做算术平均**（§9.2）。关键维度取下界 + 红线一票压顶。例：纠错维度 A 为 2、B 为 1 → 取「1-2 不稳定」并标波动，**不取均值 1.5**（均值会把「连失 5 次的循环」洗白成「基本到位」）。
- **置信度随聚合传播**。3 个 session、prompts 多为空 → 团队报告必须标「低置信，仅供人工复核」，而不是给个干净的 L2。

层级：`session map → 个人 reduce（按 §9.2/§9.3）→ 团队 reduce`。团队级**只聚合等级分布 + 共性风险，不碰原始 transcript**（守 §11 隐私边界）。

### 18.8 行为锚点标尺 + 强制极值

治 central tendency 的两个机制：

- **行为锚点标尺**：每档配真实正 / 反例（「长这样才算 3，长那样只能给 1」），禁用「基本到位 / 有一点」这类抽象词。
- **强制找极值**：每个维度必须挂**最强证据或最弱证据**，禁止写「总体平衡」式总结——总结是区分力的杀手。

沿用现有 `report-prompts.md` 的「自包含评分块」骨架（维度名 → 分数 → 紧跟直属证据链，明文禁止「分数堆顶、材料堆底」），它就是「输入对应到维度」的执行保障，不重新发明。

### 18.9 实现时最常见的 7 个错误（阶段二/三 review gate 必查）

1. **拿 AI 自述当事实**：AI 说「测试通过」就给验证维度高分。必须以工具执行记录为准，无 command 证据 = 未验证（沿用现有 Hallucination Guard）。
2. **证据不足时硬打分**：prompts 为空 / 只有闲聊 / 只 3 个 session 也输出干净 L2。必须降级为「证据不足」+ 低置信。
3. **维度求平均、忽略红线**：见 18.7。
4. **只用 prompt 打分**：奖励表演式 prompt、对真实风险失明。必须用事件流（18.2）。
5. **把任务成败当能力**：复杂任务失败但设了边界 / 喊了停 = 仍可能高分；简单任务成功但全程裸奔 = 不能高分（§6.4）。
6. **人身化判断**：写「这个人不认真」而非「未发现验证命令」（§6.2 红线）。
7. **隐私泄漏 + 证据糊成一团**：manager 报告塞完整 prompt / 敏感路径；或分数与证据分离无法复核（§11 + 自包含骨架治这个）。

### 18.10 前提与未决项

- **真·L0 弁别需要别人的 transcript。** 单个强主导者自己的数据只能拉出「风格差」，拉不出「等级差」——本机 662 个 session 验证已证实：找不到一条干净的「被 AI 带走」案例。因此「**别人 transcript 从哪进来**」(§16) 是比 A/B 范围更先要定的前置问题；在它定下来前，本评估只能在单人身上验「风格」，不能用于跨人考核。
- **缺陷追踪范围**：MVP 做 A 类（session 内露头），B 类（静默出厂）依赖外部数据接入，列为 §16 后续（见 18.5）。
